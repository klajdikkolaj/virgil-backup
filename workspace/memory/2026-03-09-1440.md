## 1) Meta Graph API prerequisites (Instagram posting)

1. **Account type**
   - You need an **Instagram professional account** (business or creator), not a personal account.
   - If you use **Instagram API with Facebook Login**, the IG account must be linked to a **Facebook Page**, and the user must have admin-equivalent Page tasks.

2. **Choose login model**
   - **Instagram Login (Business Login for Instagram):** IG credentials + `graph.instagram.com`
   - **Facebook Login for Business:** FB credentials + `graph.facebook.com` (and `rupload.facebook.com` for resumable video uploads)

3. **Permissions (minimum for publishing)**
   - Instagram Login path: `instagram_business_basic`, `instagram_business_content_publish`
   - Facebook Login path: `instagram_basic`, `instagram_content_publish`, `pages_read_engagement`, `pages_show_list`
   - Meta’s permissions reference also lists dependencies and App Review screencast requirements for `instagram_content_publish`.

4. **Access level + review**
   - **Standard Access** is for app-role users/testing/internal use.
   - **Advanced Access** is required for external/non-role client onboarding.
   - Advanced Access requires **App Review** and **Business Verification**.

5. **Token lifecycle**
   - Auth code: ~1 hour
   - Short-lived token: ~1 hour
   - Long-lived token: **60 days** (refreshable)

6. **Publishing constraints**
   - Official flow: create media container (`/{ig-id}/media`) then publish (`/{ig-id}/media_publish`).
   - JPEG-only for image publishing; some features (shopping tags, branded content tags, filters) are not supported in this flow.
   - Rate-limit docs currently show **100 API-published posts per 24h moving window**; one carousel section still mentions 50, so check `/{ig-id}/content_publishing_limit` at runtime and enforce conservatively.
   - If Page Publishing Authorization (PPA) is required on the linked Page, publishing can be blocked until completed.

## 2) Safe vs risky/forbidden automation

### Safe (recommended)
- Posting approved content to consenting, connected professional accounts via official Graph API endpoints.
- Scheduled publishing with queueing, idempotency, retries/backoff, and rate-limit checks.
- Telegram alerts for job success/failure/status only (internal reporting/control plane).

### Risky (high rejection/enforcement risk)
- Requesting broad permissions you don’t actively use.
- Running “full auto” without explicit human/client approval records.
- Near-limit bulk scheduling without preflight checks (token validity, publishing quota, container status).
- UI/browser scripting as a fallback when API can’t do something (fragile + policy risk).

### Forbidden / likely policy-violating
- Publishing without user consent.
- Spam/deceptive behavior or engagement manipulation patterns.
- Buying/selling likes/followers/comments.
- Bypassing official APIs (private/mobile reverse-engineered endpoints).

---

## 3) Recommended architecture (OpenClaw + Meta worker + Telegram bot)

**A. OpenClaw orchestration layer (control plane)**
- Receives operator/client instructions (create post, schedule, pause, retry).
- Applies business rules: approval gate, posting windows, per-client limits.
- Writes normalized jobs to a queue (`PENDING -> PROCESSING -> SUCCESS/FAILED`).

**B. Meta API worker (execution plane)**
- Pulls jobs, loads encrypted tenant credentials/tokens.
- Preflight:
  - permission scopes present,
  - token freshness/refresh,
  - account linkage checks,
  - `content_publishing_limit` check,
  - media validation (format/aspect/public URL).
- Publish pipeline:
  1. `POST /{ig-id}/media` (or resumable upload path for large video/reels)
  2. poll container `status_code`
  3. `POST /{ig-id}/media_publish`
  4. fetch media id/permalink and persist result
- Handles retryable vs non-retryable failures cleanly.
- Exposes webhook endpoint for async events where applicable.

**C. Telegram bot notifier (ops + client reporting)**
- Sends structured notifications (`sendMessage`): submitted, published, failed, retry scheduled.
- Includes job ID, account, scheduled time, API error excerpt.
- Optional inbound commands (`/status`, `/retry`, `/pause`) route back to OpenClaw orchestration.
- Prefer webhook mode for production reliability; avoid mixing webhook and polling simultaneously.

**D. Persistence/Audit**
- DB tables: clients, accounts, consent records, tokens, publish jobs, delivery logs, error logs.
- Immutable audit trail for who approved what and when.
